## El proyecto
Digital Booking es un sitio web que nos permite realizar reservas de alojamientos categorizados en hoteles, hostels, departamentos y bed & breakfast en distintos lugares turísticos del mundo.
El sitio puede ser utilizado por usuarios para buscar alojamientos y realizar reservas, y también por administradores, que además pueden publicar sus productos.

Nuestro objetivo es desarrollar un producto digital acorde a las funcionalidades requeridas por el cliente. Nuestra prioridad era lograr que sea responsivo, intuitivo y fácilmente de entender, para que los usuarios tengan una buena experiencia y vuelvan a visitarlo.

## Metodología de trabajo

Trabajamos con Metodologías Ágiles. Para ello nos organizamos siguiendo las ceremonias de SCRUM (lectura y validación del sprint, daily, weekly, review, retro), modularizando el proyecto en 4 Sprints de 2 semanas cada uno.<br>
A lo largo del proyecto contamos con un Scrum Master, un Product Owner y el apoyo de diferentes Tech Leads en caso que necesitáramos realizar consultas técnicas. Utilizamos el Boards de GitLab como herramienta de organización y asignación de tareas y Metro Retro para asegurarnos de mejorar como equipo de manera progresiva para asi lograr los objetivos. 
Desde el comienzo del proyecto nos dividimos de manera clara los roles pero también establecimos solucionar en grupo aquellas dudas y features más complejas a resolver a lo largo del desarrollo.

## Bitácora del proyecto

<details><summary>Sprint 1</summary>
En este primer sprint empezamos a planificar e inicializar los proyectos de las diferentes areas.
<br/>
<Strong>Progreso: </Strong>Utilizamos trello para la organización del grupo, y su respectiva división de tareas. Al ser la primera experiencia de trabajo en equipo bajo metodologías ágiles, nos encontrabamos con muchos nervios, nervios que con el paso de los días, y a medida que íbamos trabajando codo a codo se fueron disipando.
<br/>
<Strong>Fortalezas: </Strong>El equipo pudo definir desde el primer instante los distintos roles, ya que cada integrante tenia claro en que áreas se encontraban sus fortalezas, lo que permitio una correcta división de tareas, además el equipo estaba al tanto de como estaba el panaroma de las tareas, por lo que no hubo dudas en ningún punto del sprint
<br/>
<Strong>Debilidades: </Strong>Un miembro del equipo abandonó al principio de este sprint, por lo que se debió suplir las tareas que habìan restado. También en la retro concluimos que la estimación de los tiempos por tareas debía ser más precisa

<br/><br/>
[Link MetroRetro del Sprint 1 ](https://metroretro.io/board/LBVX9T7YGNAH)
</details>

<details><summary>Sprint 3</summary>
<Strong>Progreso: </Strong> Ingresó un nuevo miembro al grupo, que se sumó al equipo de Back-end y base de datos, permitiendo que este sprint, que estaba enfocado en el back, se realice más fluidamente. También fue la primera vez que el grupo pidió ayuda de los TL, principalmente en el back e infra, para poder realizar las tareas.

<Strong>Fortalezas: </Strong> El equipo ya estaba mejor definido, lo que permitió que las tareas se dividan correctamente, inclusive aquellas de infraestructura que estaban pendientes

<br/>
<Strong>Fortalezas: </Strong> El equipo se adaptó rápido y pudo trabajar correctamente, a pesar de no haber terminado satisfactoriamente el sprint anterior, no hubo roces ni problemas mayores, y el equipo pudo seguir su ritmo de trabajo e incluso mejorarlo.
<br/>
<Strong>Debilidades: </Strong> No encontramos debilidades específicas en este sprint, ya que al haber tenido los problemas previos, el equipo desarrollo la capacidad de comunicarse mejor, y pedir ayuda rapida a los TL de ser necesario, lo que generó que las tareas se destraben, y permitio terminar el proyecto satisfactoriamente.

## Documentación Técnica del Proyecto

### Ambiente de Desarrollo

- **Setup**

Para comenzar a trabajar, clonar el repositorio de Gitlab: 

Con SSH: 
```
$ git clone git@gl.deitech.online:ctd/proyecto-integrador-0522/0821-c1/grupo-03.git
```

Con HTTPS: 
```
$ git clone https://gl.deitech.online/ctd/proyecto-integrador-0522/0821-c1/grupo-03.git
```

Para correr el proyecto, instale localmente las siguientes librerías:

```
$ cd ../proyecto/FrontEnd
$ npm install
$ npm start
```

- **Workflow**

[Link al Workflow](https://gl.deitech.online/ctd/proyecto-integrador-0522/0821-c1/grupo-03/-/network/development)
<br>
<p>El proyecto se lleva a cabo en la rama "main". De esta rama creamos la rama "development", donde ejecutamos los test correspondientes. De ella se desprenden ramas con cada feature que se encuentra en desarrollo (las mismas están detalladas en el board de "Issues"). Una vez realizada la feature, se mergea con la rama development haciendo un Pull Request (incluyendo un mensaje descriptivo sobre el contenido del pull request). Una vez realizados los test hacemos el merge de la rama development con la rama main</p>
